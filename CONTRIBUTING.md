# 贡献指南

感谢你关注这个项目。

本文档用于说明如何在本仓库中提交问题、发起改动和整理贡献内容，帮助协作者保持一致的开发和沟通方式。

## 开始之前

- 请先阅读 `README.md` 或 `README.en.md`，了解项目目标、目录结构和运行方式。
- 如果你的改动涉及使用方式、环境变量或工作流行为，请同步更新相关文档。
- 如果你的改动较大，建议先提交 `Issue` 讨论方案，再开始实现。

## 开发流程

1. Fork 本仓库并创建分支。
2. 在本地完成开发、验证和文档更新。
3. 使用清晰的提交说明整理变更。
4. 发起 Pull Request，并补充变更背景和验证结果。

推荐分支命名：

- `feat/<short-name>`
- `fix/<short-name>`
- `docs/<short-name>`
- `refactor/<short-name>`

## 提交规范

建议提交信息尽量简洁明确，能够直接说明本次改动目的。

示例：

```text
feat: add local KB configuration docs
fix: handle empty query in qa graph
docs: improve README screenshot section
```

## Pull Request 建议内容

发起 PR 时，建议至少说明以下信息：

- 变更背景
- 主要改动
- 是否影响配置或使用方式
- 本地验证方式
- 是否补充或更新了文档

## 代码与文档要求

- 保持目录职责清晰，不要把不相关逻辑混在一起。
- 新增配置项时，请同步更新 README。
- 新增功能时，优先考虑是否需要补充测试。
- 不要提交密钥、令牌、账号密码或本地私有路径。
- 若改动会影响已有流程，请在 PR 中明确说明兼容性影响。

## 测试建议

当前项目测试以 `unittest` 为主，可在仓库根目录执行：

```bash
python -m unittest discover -s tests
```

如果改动涉及桌面端，也建议至少说明：

- 启动方式
- 手动验证步骤
- 观察到的结果

## 文档贡献

欢迎对以下内容进行持续改进：

- `README.md` / `README.en.md`
- 使用示例
- 截图和演示说明
- 架构说明
- 常见问题

如果你新增了截图，建议统一放在 `docs/images/` 目录下。

## 问题反馈

提交 Issue 时，请尽量提供：

- 运行环境
- 复现步骤
- 实际结果
- 预期结果
- 日志、截图或报错信息

## 许可证说明

当前仓库尚未提供明确的 `LICENSE` 文件。在许可证明确之前，请在分发、商用或二次发布前与仓库维护者确认使用范围。
